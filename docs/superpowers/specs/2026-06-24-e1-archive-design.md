# E1 — Archive (soft-delete for projects & features) — design

Date: 2026-06-24
Status: approved (brainstorm), pending spec review
Epic: E1 — Multi-project gaps (`BACKLOG.md`). This slice = **archive / soft-delete**, closing the current hard-delete-only data-loss risk. Sibling E1 gaps (share-polish, invite-accept UI) are separate.

## Goal

Replace hard-delete of projects and features with a restorable soft-delete (archive). Archived items are hidden from active lists, read-only, and restorable. Permanent deletion happens only from a trash/archive view, behind an explicit confirm. No scheduler (no auto-purge).

## Decisions (locked in brainstorm)

- **Archive replaces delete.** The primary "Delete" action becomes "Archive". `DELETE` endpoints are repurposed to **permanent purge**, allowed only when the item is already archived.
- **Manual purge only.** Archived items live in trash indefinitely; an explicit "Delete permanently" hard-deletes (still cascades via FKs). No auto-purge / cron.
- **Scope:** `projects` + `features`.
- **Permissions:** project archive/restore/purge = **owner**; feature archive/restore = **editor**; feature purge = **owner**.

## §1 Data — migration 0017

Add a nullable `archivedAt timestamptz` to both tables, mirroring the existing `templates.archivedAt` idiom (`schema.ts:225`):

```ts
// projects table
archivedAt: timestamp('archived_at', { withTimezone: true }),
// features table
archivedAt: timestamp('archived_at', { withTimezone: true }),
```

`NULL` = active; non-null = archived (value is the archive timestamp). No new tables. Migration generated by drizzle-kit (`pnpm --filter @productmap/db generate` → `0017_*.sql`).

## §2 API

### Projects (`apps/api/src/routes/projects.ts`)

- `POST /:projectId/archive` — `requireMembership('owner')`. Sets `archivedAt = now()` when currently active; idempotent-safe (no-op/200 if already archived). Returns `{ id, archivedAt }`.
- `POST /:projectId/restore` — `requireMembership('owner')`. Sets `archivedAt = null`. 404 if project absent.
- `DELETE /:projectId` — **repurposed to permanent purge.** `requireMembership('owner')`. Allowed **only when the project is already archived** (`archivedAt IS NOT NULL`); otherwise **409** `{ error: 'not_archived' }`. Performs the existing hard delete (FK cascade to features/docs/notifications/etc.).

### Features (`apps/api/src/routes/features.ts`)

- `POST /:id/archive` — editor+ (matches current feature-mutation gate). Sets `archivedAt = now()`.
- `POST /:id/restore` — editor+. Sets `archivedAt = null`.
- `DELETE /:id` — **repurposed to permanent purge.** Owner-only. Allowed only when the feature is already archived; else **409** `{ error: 'not_archived' }`. Existing pre-delete cleanup (votes, collaborators) + hard delete retained.

### List scoping — hide archived from active views

Default display lists filter `archivedAt IS NULL`. A dedicated archived query returns the trash set.

- **Projects:** `GET /projects` (`projects.ts:23`) returns active only (`archivedAt IS NULL`). Add a `?archived=1` query param to the same route returning the caller's archived projects (membership-scoped) instead. Dashboard project rollup (`dashboard.ts:62`) filters active.
- **Features (primary display lists only — see boundary):** board (`features.ts:66`), single-project overview (`overview.ts:24`), releases feature rollup (`releases.ts`), dashboard `myWork`/feature queries (`dashboard.ts:76`), document feature picker (`documents.ts:226`) filter `isNull(features.archivedAt)`. The board features route (`GET /projects/:projectId/features`) gains a `?archived=1` param returning that project's archived features instead.

**Boundary (deliberate):** id-based feature lookups and analysis paths (`copilot.ts`, `plans.ts`, `deps.ts`, `ai.ts`) are **not** filtered — they operate on explicit ids or whole-project analysis, are read-only, and an archived feature resolving there is harmless. Filtering every feature query is out of scope for this slice; the primary user-facing lists are covered.

### Read-only enforcement on archived projects

Writes to an archived project are rejected. Enforce in **one place**: the project-scoped content mount (`projectScopedContent` / its membership wrapper). After membership resolves, if the project's `archivedAt IS NOT NULL` and the request method is a write (`POST/PATCH/PUT/DELETE`), return **409** `{ error: 'project_archived' }`. This blocks all content writes (features, docs, comments, etc.) under an archived project with a single guard. The project **management** routes (archive/restore/purge/members) live on `projectsRoutes`, not the content mount, so restore/purge are unaffected. Reads of an archived project remain allowed (needed to render the trash view + restore).

## §3 Web (`apps/web`)

- **Delete → Archive relabel.** Existing project/feature "Delete" controls become "Archive" (soft). The destructive confirm dialog copy changes from permanent-delete to archive (restorable).
- **Archived projects view.** An "Archived" section reachable from the dashboard/project switcher (collapsible list or a `/app` sub-view) lists the caller's archived projects with **Restore** and **Delete permanently** (typed/confirm guard reusing the existing destructive-confirm component). Hidden from the main switcher/dashboard list.
- **Archived features view.** Per-project: archived features hidden from board/roadmap; a small **"Archived"** view/filter on the board lists them with Restore / Delete permanently.
- **Hooks** (`apps/web/src/lib/api.ts`): `useArchiveProject`/`useRestoreProject`/`usePurgeProject`, `useArchiveFeature`/`useRestoreFeature`/`usePurgeFeature`, `useArchivedProjects`, `useArchivedFeatures`. Mutations invalidate the relevant active + archived lists (mirror existing optimistic patterns where cheap).

## §4 Testing (mirror existing route + component test patterns)

API:
- archive sets `archivedAt`, hides project from `GET /projects` (active) and surfaces it in the archived query.
- restore clears `archivedAt`, returns it to active lists.
- purge: `DELETE` on an **active** project/feature → 409 `not_archived`; on an **archived** one → hard-deletes (and FK cascade confirmed: features/docs gone).
- read-only: any content write (e.g. create feature, post comment, patch doc) under an archived project → 409 `project_archived`; the same write under an active project still 201/200.
- archived features filtered from board/overview; present in the archived-features query.
- permissions: non-owner archive/restore/purge project → 403; viewer archive feature → 403; editor feature purge → 403 (owner-only).

Web:
- archived project appears in the archived view with Restore + Delete-permanently; restore moves it back; the main list excludes archived.
- "Delete" control now reads "Archive" and calls the archive mutation (not purge).

## Out of scope (later)

- Auto-purge / retention scheduler.
- Archiving docs/ideas/releases individually (project-archive hides them transitively; feature-archive covers the feature surface).
- Filtering archived features from AI/copilot/deps/plans analysis paths.
- Share-polish and invite-accept-UI (separate E1 slices).
